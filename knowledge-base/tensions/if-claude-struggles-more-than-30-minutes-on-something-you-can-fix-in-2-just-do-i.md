**Side A:** "If Claude struggles more than 30 minutes on something you can fix in 2, just do it yourself"
**Side B:** "It IS faster to do it yourself the first time — but accepting that upfront cost is the only path to compounding leverage"
**Reason:** Both address when to do things yourself vs delegate. One optimizes for short-term efficiency (the 30-minute rule), the other for long-term leverage (accept short-term cost to build compounding delegation muscle). Sometimes stepping in saves wasted cycles, but habitually doing so prevents building 10x output systems.
**Resolution:** The key variable is whether the task is a one-off or a recurring pattern. Side A (30-minute rule) is correct for one-off tasks: environment-specific issues, config quirks, visual CSS tweaks -- tasks that will never repeat in the same form, where stepping in saves pure waste. Side B (accept upfront cost for compounding leverage) is correct for recurring patterns: if Claude will encounter this class of problem repeatedly, investing time to teach it (via CLAUDE.md rules, better prompts, workflow documentation) compounds into future time savings. The mistake is applying the delegation mindset to genuinely one-off tasks (wasting time teaching something that will not recur) or applying the efficiency mindset to recurring patterns (saving 28 minutes today but losing hundreds of hours of future leverage). The practical test: will this exact class of problem arise again? If yes, invest in delegation. If no, step in.

**Prediction:** Developers who categorize Claude Code tasks as one-off vs recurring before deciding whether to intervene will report higher overall productivity than those who apply either the 30-minute rule or the delegation principle uniformly.
