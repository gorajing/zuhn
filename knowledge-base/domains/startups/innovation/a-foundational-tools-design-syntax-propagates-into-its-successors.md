C was created in 1972. Its curly brace blocks, semicolons, pointer notation, and function structure were engineering choices made for Unix development. Yet Java in 1995, JavaScript in 1995, PHP in 1994, and many others adopted C's surface syntax with minimal deviation. The pattern was so established that departing from it felt exotic.

This is path dependency at the language level: once C's syntax was adopted by C++ (which dominated enterprise development), every successor language that wanted C++ developer adoption had to look similar. Python and Ruby are notable exceptions that paid the adoption cost of looking different.

The principle generalizes across domains: when a foundational platform achieves critical mass, subsequent entrants face a choice between compatibility (easier adoption, harder differentiation) and incompatibility (harder adoption, possible differentiation). Most choose compatibility. The original designer's choices become constraints on the entire subsequent ecosystem.
