Cheung warns against the 'Frankenstein approach': users give you feedback, you build every feature they suggest, and the product becomes a stitched-together monster that doesn't coherently solve anything. The antidote is to treat every feature request as a symptom, not a diagnosis.

When a user says 'build X,' the real signal is: 'I have a problem that X might solve, but X might not be the best solution.' The founder's job is to dig into the why — is this a problem the product created? Is it a prerequisite the user needs before they'll pay? The underlying problem often has a simpler, more elegant solution than the feature the user suggested. This requires discipline because building the requested feature feels productive and makes the user happy in the short term, but it accumulates complexity that makes the core product harder to understand and iterate on.
