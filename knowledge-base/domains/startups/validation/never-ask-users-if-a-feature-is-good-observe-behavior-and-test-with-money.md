Shear warns against the most common user interview trap: describing a feature and asking if it's good. Users reliably say 'oh yeah, that's great' to proposed features, but when you build them, nobody switches to get them. This creates a false confidence loop where founders spend months building validated-sounding features that nobody actually wants.

The alternative validation methods Shear recommends are ordered by reliability. Best: get users to commit money upfront ('if you're not $5 excited about it, you're probably not very excited about it'). Second best: hack together a minimal version and put it in front of users (e.g., a browser extension on top of Google Docs rather than rebuilding the whole app). Worst but still better than asking: draw the workflow and walk users through it. The key principle is that stated preferences diverge massively from revealed preferences, and the cheapest way to reveal true preferences is the money test.
