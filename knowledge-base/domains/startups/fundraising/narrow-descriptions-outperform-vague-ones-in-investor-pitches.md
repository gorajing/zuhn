Founders resist narrow descriptions because early-stage products have many possible futures. But vague descriptions ('a system to collaboratively leverage information') register as meaningless boilerplate. Investors discard them and wait impatiently for something concrete.

The counterintuitive insight is that a description which feels misleadingly narrow ('an easy web-based database') actually works better as a starting point. It gives investors a concrete mental model to build on. You can then flesh it out through successive approximation — like incremental software development where you always have working code. In this case, 'working code' means a working description in the investor's head. The narrower you start, the faster they understand, and the sooner they can evaluate whether it's worth pursuing.
