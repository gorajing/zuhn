Founders resist concise descriptions because early-stage startups have many possibilities, and narrow descriptions feel misleading. But Paul Graham argues this is backwards: the calculus limit of a description that could be anything is zero content. Investors will discard vague sentences as meaningless boilerplate.

The correct approach mirrors incremental development: start with a gripping but perhaps overly narrow description, then flesh it out. At every point you should have a 'working description' in the investor's head — the same way you should always have working code. 'We built an easy web-based database' beats 'a system to allow people to collaboratively leverage the value of information' every time.
