Paul Graham observes that as a startup description approaches 'could be anything,' its information content approaches zero. Founders resist narrow descriptions because they feel misleadingly limited, but investors process pitches serially and will discard boilerplate like 'collaboratively leverage the value of information' without retaining anything.

The fix is to treat your pitch like incremental development: start with a gripping but perhaps overly narrow description ('easy web-based database'), then flesh it out. At every point you need a 'working description' in the investor's head. This mirrors how good software ships — a working prototype first, features second. The goal isn't to describe everything your system might become, but to be compelling enough to earn the next conversation.
