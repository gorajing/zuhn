Founders resist narrow descriptions because early-stage startups have many possibilities. But Paul Graham argues this is precisely the wrong instinct: the more general your description, the less memorable and useful it becomes. 'A system to allow people to collaboratively leverage the value of information' conveys nothing, while 'an easy web-based database' is immediately graspable.

The correct approach mirrors incremental development: start with a description that's gripping but perhaps overly narrow, then flesh it out. At every point you need a 'working description' in the investor's head — the same way you need working code at every stage of development. Your goal isn't to describe everything the system might become, but to be compelling enough to earn the next conversation.
