A recurring pattern in CS50 Python is Malan writing a naive, ugly-but-working solution first, then progressively refactoring it — extracting functions, adding error handling, improving naming, introducing classes. He explicitly teaches students to resist the urge to write perfect code on the first pass. This 'make it work, make it right, make it fast' progression (attributed to Kent Beck) is demonstrated dozens of times across the course.

This maps directly to startup execution. Founders who try to architect the perfect system before proving the concept works are engaging in premature optimization — spending design calories on code that might be thrown away entirely. The CS50 approach mirrors the lean startup loop: build the simplest thing that could work, validate it works, then invest in quality. The psychological benefit is equally important: a working ugly solution provides momentum and confidence, while a half-finished elegant solution provides neither. Correctness is a precondition for elegance, not a competitor with it.
