Throughout CS50 Python, Malan deliberately writes buggy code, runs it, shows the error, and then fixes it — often multiple times per lecture. This isn't accidental or inefficient; it's a core pedagogical strategy. Students learn to read tracebacks, recognize common failure patterns (off-by-one, type mismatches, scope confusion), and develop the meta-skill of systematic debugging.

This works because debugging is pattern recognition, and pattern recognition requires exposure to the patterns. A student who has only seen correct code will freeze at their first NameError. A student who has watched the instructor hit that error twenty times, read the traceback, and trace it back to a misspelled variable name has already built the neural pathway for that recovery. Error-driven learning also normalizes failure as part of the process rather than evidence of incompetence — a psychological shift that separates persistent learners from those who quit at the first wall.
