Malan's teaching method has a consistent structure: introduce a concept with a naive implementation that works, then show how it breaks with edge cases, then fix it, then show how the fix introduces new problems, then fix those. The calculator example is paradigmatic: 1+2 produces '12' (string concatenation), which leads to learning about type conversion, which leads to learning about error handling for non-numeric input. Each 'failure' is actually the learning event.

This maps to a deeper cognitive principle: understanding is not transmitted, it's constructed through resolving contradictions between expectation and reality. The student who expects 1+2=3 and sees '12' has a productive confusion that, once resolved, creates durable understanding of type systems. This is why worked examples in textbooks are less effective than struggling with problems — the struggle IS the learning mechanism, not an impediment to it. The implication for any learning context is that the optimal difficulty level is one where learners encounter frequent, resolvable surprises.
