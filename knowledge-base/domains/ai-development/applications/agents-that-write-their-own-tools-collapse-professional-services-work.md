The speakers trace an evolution: chatbots you can talk to, then chatbots with tools, and now a tool you give the chatbot that can make other tools it doesn't have. Because a coding agent operates through the command line and files — the universal pre-GUI interface to a computer — anything expressible as commands and files is in scope, including non-coding work like data analysis over CSV folders.

The business consequence they highlight: you can build enterprise software that writes its own plug-in connectors to each customer's API on the spot. That is work a professional-services team used to do manually for every account. One presenter demoed a Kanban board that fixes its own bugs to make the point that the software can 'talk back to itself.'

Marked inspiration and time-sensitive because it's a directional bet on where the SDK and model capabilities are heading (longer-horizon, unsupervised tasks; a rising 'trust ceiling') rather than a settled, fully-reliable pattern today. The durable idea worth carrying forward: when integration work is the cost center, an agent that can author its own integrations changes the unit economics of serving each new customer.
